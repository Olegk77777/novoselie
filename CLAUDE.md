## Что мы делаем
Браузерная казуальная игра: обустройство квартиры в эстетике пост-советских 90-х. Полный концепт — в файле CONCEPT.md (прочитай его первым делом). Текущая цель — версия v0.1 по плану из концепта.

## Технологии (НЕ менять без обсуждения)
- Чистый HTML + JavaScript + Three.js (подключаем через CDN, без сборщиков/npm на старте)
- Один запуск: открыть index.html через локальный сервер (`python3 -m http.server` — объясни Олегу как)
- Изометрическая камера, зафиксирована, поворот только шагами по 90°
- Сетка на полу, предметы ставятся по клеткам
- Низкополигональные модели собираем КОДОМ из примитивов Three.js (BoxGeometry и т.п.) — стиль PS1
- Текстуры Олег кладёт в папку /textures (генерирует сам) — натягиваем на модели

## Архитектура (железные правила)
1. **Предметы = данные, не код.** Все предметы описаны в items.json: id, размер в клетках, очки уюта, можно ли на стену/пол, текстура. Новый предмет = новая запись в JSON, без правки логики.
2. **Локализация с первого дня.** ВСЕ тексты игры — только в locales/ru.json (потом добавим en.json). В коде — только ключи. Ни одной русской строки внутри .js файлов.
3. **Квесты = данные.** Условия квестов описываем декларативно в quests.json (тип условия + параметры), движок их проверяет.
4. **Никаких реальных брендов** — названия, логотипы, узнаваемые товары только выдуманные.

## Текстуры — как ставить задачу Олегу (железное правило)
Текстуры генерирует Олег сам. Каждый раз, когда нужна текстура, Клод даёт задание в таком формате:
- **Имя файла и путь:** например `textures/floor_parquet.png` (только латиница, нижний регистр, подчёркивания)
- **Размер:** степень двойки, по умолчанию 512×512 (большие поверхности — 1024×1024)
- **Формат:** PNG (или JPG, если без прозрачности и важен вес)
- **Бесшовность:** обязательно указывать, нужна ли seamless/tileable текстура
- **Промпт для генерации:** готовый, на английском, без параметров-флагов (--ar и т.п.)

Пока текстуры нет — модель работает с простым цветом-заглушкой, игра не ждёт.

Файлы от Олега перед коммитом приводить в порядок: проверять имя (пробелы и двоеточия из путей вылезают в имя файла), сжимать до рабочего размера (`sips -z 1024 1024 -s format jpeg -s formatOptions 80 ...`), цель — не больше ~500 КБ на текстуру. Оригиналы не удалять, а переносить в ~/Downloads.

## Технические решения (зафиксировано, не менять молча)
- **Three.js 0.170.0**, подключён через importmap в index.html (CDN unpkg). Версию не обновлять без причины.
- **Запуск игры:** из папки проекта `python3 serve.py`, потом открыть http://localhost:8123 в браузере. serve.py — это http.server с заголовками no-cache, чтобы браузер не показывал старые версии файлов после правок. Просто двойной клик по index.html НЕ работает (браузер блокирует загрузку модулей и JSON с диска).
- **Кэш / версионирование:** браузер (особенно на GitHub Pages) кэширует JS-модули и текстуры. Локально это лечит serve.py. Для Pages у локальных импортов стоит `?v=N` (только в index.html и game.js — больше нигде локальных импортов нет). При ЛЮБОМ изменении кода перед коммитом поднимать версию на 1: заменить текущее число на следующее во всех `?v=` в index.html и src/game.js, например `sed -i '' 's/?v=СТАРОЕ/?v=НОВОЕ/g' index.html src/game.js` (подставить реальные числа, напр. 3→4). НЕ запускать sed по этому файлу (CLAUDE.md) и комментариям — там примеры с числами, они затрутся. Текстуры не версионированы — если Олег заменит текстуру тем же именем, нужен hard-refresh (Ctrl+Shift+R).
- **Система координат:** 1 клетка сетки = 1 юнит Three.js. Пол центрирован в начале координат (сетка 10×8 → x от −5 до 5, z от −4 до 4). Центр клетки (col,row): x = −cols/2 + col + 0.5, z = −rows/2 + row + 0.5. Формулы — в src/grid.js.
- **Камера:** ортографическая (изометрия), позиция (10,10,10), смотрит в (0,0,0). Логика вынесена в src/camera.js. Зафиксированы РАКУРС и ПОВОРОТ (поворот будет шагами по 90° позже) — а вот МАСШТАБ менять можно и нужно: камера сама вписывает комнату в экран (авто-fit под любой размер окна), плюс ручной зум колесом мыши и щипком двумя пальцами (pinch) на сенсоре. Менять угол/вращать свободно — нельзя.
- **Референс планировки:** Planirovka.png в корне — образ всей квартиры на будущее (v0.3+), в v0.1 строим одну комнату.
- **Стены:** src/walls.js — только две дальние стены (ближние не рисуем, чтобы видеть внутрь). Позиции окна и двери — константы WINDOW/DOOR в walls.js; при переходе к выбору планировок переедут в data/layouts.json. getWallSurfaces() отдаёт описание лицевых поверхностей (плоскость, ось «вдоль», вырезы окна/двери) для настенного размещения.
- **Настенное размещение:** предметы с placement:"wall" вешаются на стену (placement.js, wall-режим). Призрак прилипает к ближайшей стене под курсором (raycast по плоскостям), двигается с шагом 0.5 по «along»/высоте, поворачивается под стену, краснеет на вырезах (окно/дверь) и за границами. UI-состояние 'placingWall' (своя подсказка, без кнопки поворота).
- **Предметы:** data/items.json (поля описаны в _comment внутри файла) + реестр моделей MODEL_BUILDERS в src/items.js (ключ = поле model). Ковры — layer: "rug": отдельный слой занятости, мебель ставится поверх. Настенные предметы (placement: "wall") пока отключены в панели — механика стен будет отдельным шагом.
- **Расстановка:** шаг перемещения — ПОЛклетки (SUB=2 в placement.js), поворот — по 45° (8 шагов; диагональные предметы занимают описанный прямоугольник, округлённый вверх). Поверхности: у предмета с surfaceHeight на верх можно ставить предметы с mountable:true (ТВ и магнитофон на тумбу/стол/табурет); связь хозяин↔гость в userData (occupant/host), клик по занятому хозяину сначала берёт гостя. Модели mountable-предметов строить с низом на y=0.
- **Текстуры материалов:** items.js → texturedMaterial(): wood_light.jpg (дерево), plastic_dark.jpg (пластик), metal_brushed.jpg (металл), rug_pattern.jpg (узор ковра, маппится 1:1 без повтора). Нет файла — работает цвет-заглушка.
