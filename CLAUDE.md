# CLAUDE.md — проект «Новоселье» (Housewarming)

## Кто заказчик
Олег — вайб-кодер, НЕ программист. Объясняй каждый шаг простым языком, как новичку: что делает команда, зачем нужен файл, что произойдёт. Никакого жаргона без расшифровки. Все ответы — на русском.

## Что мы делаем
Браузерная казуальная игра: обустройство квартиры в эстетике пост-советских 90-х. Полный концепт — в файле CONCEPT.md (прочитай его первым делом). Текущая цель — версия v0.1 по плану из концепта.

## Эстетика и атмосфера (железное правило — определяет ВСЕ решения)
Игра — **думерская** по духу. Это не оформление, а суть продукта: пост-советская меланхолия, брутализм, спальные районы, 1990-е, панельная тоска, вечный ноябрь, уютная зима, тёплая аналоговая печаль. Камертон — «Молчат Дома» и весь нынешний западный интерес к doomer / Eastern-bloc nostalgia / brutalism. Это ещё и **рыночное окно**: на этой волне игра может выстрелить, поэтому атмосферу не размываем.

Каждое решение — модель, текстура, цвет, свет, текст, звук, UI, название — проверяй вопросом: **работает ли это на думерскую атмосферу?** Если деталь делает картинку ярче/мультяшнее/«казуальнее» в ущерб настроению — переделать.

Ориентиры на практике:
- **Палитра** приглушённая: серый бетон, охра, бордо, тёплое дерево, ламповый свет в холодных сумерках. Не кислотные цвета, не глянец.
- **Свет и время** — вечер/ночь, тёплая лампа в холодной комнате. За окном — панельки, редкие огни, провода, вечные сумерки (уже сделано шейдером).
- **Предметы** — аналоговые и узнаваемо-советские: ковёр на стене, хрусталь в серванте, ламповый ТВ с антеннами, бобинник/магнитола, торшер. Тёплый тлен, не новодел.
- **Тексты** — авторский голос Олега (скилл `oleg-writing-voice`): меланхолия через предмет, без пафоса, с тихой теплотой. Не казённый игровой UI-язык там, где можно живой.
- **Звук (когда дойдём)** — пост-панк/синтвейв/эмбиент в духе канала «Пыль на кассете», аналоговый шум, не бодрая казуальная музыка.

Меланхолия — тёплая, не давящая: это про уют и тихую грусть новоселья, а не про хоррор. Игрок должен почувствовать «дом», а не «развалину».

## Технологии (НЕ менять без обсуждения)
- Чистый HTML + JavaScript + Three.js (подключаем через CDN, без сборщиков/npm на старте)
- Один запуск: открыть index.html через локальный сервер (`python3 -m http.server` — объясни Олегу как)
- Изометрическая камера, зафиксирована, поворот только шагами по 90°
- Сетка на полу, предметы ставятся по клеткам
- Низкополигональные модели собираем КОДОМ из примитивов Three.js (BoxGeometry и т.п.) — стиль PS1
- Текстуры Олег кладёт в папку /textures (генерирует сам) — натягиваем на модели

## Архитектура (железные правила)
1. **Предметы = данные, не код.** Все предметы описаны в items.json: id, размер в клетках, очки уюта, можно ли на стену/пол, текстура. Новый предмет = новая запись в JSON, без правки логики.
2. **Локализация с первого дня.** ВСЕ тексты игры — только в locales/ru.json (потом добавим en.json). В коде — только ключи. Ни одной русской строки внутри .js файлов.
3. **Квесты = данные.** Условия квестов описываем декларативно в quests.json (тип условия + параметры), движок их проверяет.
4. **Никаких реальных брендов** — названия, логотипы, узнаваемые товары только выдуманные.

## Текстуры — как ставить задачу Олегу (железное правило)
Текстуры генерирует Олег сам. Каждый раз, когда нужна текстура, Клод даёт задание в таком формате:
- **Имя файла:** давать ПРОСТЫМ текстом в backticks, отдельной строкой, чтобы легко выделить и скопировать — НЕ markdown-ссылкой `[...](...)`. Только латиница, нижний регистр, подчёркивания. Пример: `plastic_dark.png`. Папку (`textures/`) упоминать словами рядом, не в имени.
- **Размер:** степень двойки, по умолчанию 512×512 (большие поверхности — 1024×1024). Цельные рисунки (ковёр) могут быть прямоугольными под пропорции предмета.
- **Формат:** PNG (или JPG, если без прозрачности и важен вес)
- **Бесшовность:** обязательно указывать, нужна ли seamless/tileable текстура
- **Промпт для генерации:** готовый, на английском, без параметров-флагов (--ar и т.п.)

Пока текстуры нет — модель работает с простым цветом-заглушкой, игра не ждёт.

Файлы от Олега перед коммитом приводить в порядок: проверять имя (пробелы и двоеточия из путей вылезают в имя файла), сжимать до рабочего размера (`sips -z 1024 1024 -s format jpeg -s formatOptions 80 ...`; для прямоугольных — `sips -Z МАКССТОРОНА` сохраняет пропорции), цель — не больше ~500 КБ на текстуру. Оригиналы не удалять, а переносить в ~/Downloads.

**Negative cache:** если код уже ссылался на текстуру, которой не было (сервер отдавал 404), браузер запоминает 404. После добавления файла нужен hard-refresh (Ctrl+Shift+R; на планшете — потянуть страницу/очистить кэш), иначе текстура не появится, хотя файл на месте. Локально serve.py это лечит, но уже открытую вкладку всё равно обновить жёстко.

## Технические решения (зафиксировано, не менять молча)
- **Three.js 0.170.0**, подключён через importmap в index.html (CDN unpkg). Версию не обновлять без причины.
- **Запуск игры:** из папки проекта `python3 serve.py`, потом открыть http://localhost:8123 в браузере. serve.py — это http.server с заголовками no-cache, чтобы браузер не показывал старые версии файлов после правок. Просто двойной клик по index.html НЕ работает (браузер блокирует загрузку модулей и JSON с диска).
- **Кэш / версионирование:** браузер (особенно на GitHub Pages) кэширует JS-модули и текстуры. Локально это лечит serve.py. Для Pages у локальных импортов стоит `?v=N` (только в index.html и game.js — больше нигде локальных импортов нет). При ЛЮБОМ изменении кода перед коммитом поднимать версию на 1: заменить текущее число на следующее во всех `?v=` в index.html и src/game.js, например `sed -i '' 's/?v=СТАРОЕ/?v=НОВОЕ/g' index.html src/game.js` (подставить реальные числа, напр. 3→4). НЕ запускать sed по этому файлу (CLAUDE.md) и комментариям — там примеры с числами, они затрутся. Текстуры не версионированы — если Олег заменит текстуру тем же именем, нужен hard-refresh (Ctrl+Shift+R).
- **Система координат:** 1 клетка сетки = 1 юнит Three.js. Пол центрирован в начале координат (сетка 10×8 → x от −5 до 5, z от −4 до 4). Центр клетки (col,row): x = −cols/2 + col + 0.5, z = −rows/2 + row + 0.5. Формулы — в src/grid.js.
- **Камера:** ортографическая (изометрия), позиция (10,10,10), смотрит в (0,0,0). Логика вынесена в src/camera.js. Зафиксированы РАКУРС и ПОВОРОТ (поворот будет шагами по 90° позже) — а вот МАСШТАБ менять можно и нужно: камера сама вписывает комнату в экран (авто-fit под любой размер окна), плюс ручной зум колесом мыши и щипком двумя пальцами (pinch) на сенсоре. Менять угол/вращать свободно — нельзя.
- **Референс планировки:** Planirovka.png в корне — образ всей квартиры на будущее (v0.3+), в v0.1 строим одну комнату.
- **Стены:** src/walls.js — только две дальние стены (ближние не рисуем, чтобы видеть внутрь). Позиции окна и двери — константы WINDOW/DOOR в walls.js; при переходе к выбору планировок переедут в data/layouts.json. getWallSurfaces() отдаёт описание лицевых поверхностей (плоскость, ось «вдоль», вырезы окна/двери) для настенного размещения.
- **Настенное размещение:** предметы с placement:"wall" вешаются на стену (placement.js, wall-режим). Призрак прилипает к ближайшей стене под курсором (raycast по плоскостям), двигается с шагом 0.5 по «along»/высоте, поворачивается под стену, краснеет на вырезах (окно/дверь) и за границами. UI-состояние 'placingWall' (своя подсказка, без кнопки поворота).
- **Предметы:** data/items.json (поля описаны в _comment внутри файла) + реестр моделей MODEL_BUILDERS в src/items.js (ключ = поле model). Ковры — layer: "rug": отдельный слой занятости, мебель ставится поверх. Настенные предметы (placement: "wall") пока отключены в панели — механика стен будет отдельным шагом.
- **Расстановка:** шаг перемещения — ПОЛклетки (SUB=2 в placement.js), поворот — по 45° (8 шагов; диагональные предметы занимают описанный прямоугольник, округлённый вверх). Поверхности: у предмета с surfaceHeight на верх можно ставить предметы с mountable:true (ТВ и магнитофон на тумбу/стол/табурет); связь хозяин↔гость в userData (occupant/host), клик по занятому хозяину сначала берёт гостя. Модели mountable-предметов строить с низом на y=0.
- **Текстуры материалов:** items.js → texturedMaterial(): wood_light.jpg (дерево), plastic_dark.jpg (пластик), metal_brushed.jpg (металл), rug_pattern.jpg (узор ковра, маппится 1:1 без повтора). Нет файла — работает цвет-заглушка.
- **Свет и поверхности (src/lighting.js) — эстетика Хоппера / Limbo-в-цвете (железный курс по запросу Олега):** свет — ГЛАВНАЯ составляющая атмосферы (Олег фотограф). НЕ заливать комнату тёплым светом «лампочки сверху» — тёплого верхнего ключа НЕТ. Основное заполнение идёт ОТ ОКНА (холодный/сумеречный свет), базовый полумрак — синеватый и глубокий, но не чёрный; тёплый свет игрок зажигает сам (торшер/ТВ/аквариум) и он КОНТРАСТИРУЕТ с холодной мглой. Всё в `createLighting(scene)`: **WINDOW** (главный свет/заполнение, падает из окна, ЕДИНСТВЕННЫЙ кастует тень — длинные хопперовские тени; направление ФИКСИРОВАНО, чтобы shadow map была статична, а цвет/яркость реактивны), **HEMI** (рассеянное холодновато-синеватое заполнение, мягко покрывает ВСЮ комнату — киношный нейтральный fill «за кадром», низ полусферы поднят, чтобы доставало до затенённых граней; основной мягкий свет, чтобы тени не проваливались в чёрное), **FILL** (нейтральный отскок спереди, без тени), **DOOR** (мягкий ТЁПЛЫЙ конус-прожектор из дверного проёма = свет из коридора квартиры; второй тёплый якорь, контраст холодному окну; без тени; позиция из game.js — doorX/doorZ). `lighting.update(time, hasWindow)` в кадровом цикле game.js пересчитывает цвет и яркость WINDOW и яркость HEMI из ВРЕМЕНИ окна (день холодный/яркий → закат янтарь → ночь почти тьма + серебро в полнолуние → дождь свинцовый). Тёплый ключ не вернуть «по умолчанию» — это сломает курс. **SYNC:** формулы суток/сезона/погоды/полнолуния в lighting.js — ручной JS-двойник GLSL из walls.js (величины живут только в GPU); правишь оконный шейдер — правь формулы и в lighting.js (там стоит маркер SYNC). Троттлинг раз в 4 кадра.
- **Свет приборов:** торшер/ТВ/аквариум/магнитола светят точечно (`makeApplianceLight()` в items.js, PointLight без тени, с distance+decay), горят ТОЛЬКО при токе (`g.userData.powered`, гасятся в их `tick`). Свет — child модели (едет/крутится с предметом), но ВЫРЕЗАЕТСЯ из иконки (icon.js) и призрака (placement.makeGhost) обходом `o.isLight`, иначе засветит превью/«лампу в руке». Магнитола на iPad (`LOW_END`) выключена ради перфа.
- **Рендерер света:** тени — `shadowMap.enabled`, `PCFSoftShadowMap`, `autoUpdate=false` + `bumpShadows()` (needsUpdate) при перестановке/ремонте/мусоре (тень статична между ними — экономия iPad). Тон — `ACESFilmicToneMapping`, exposure 1.0 (киношный roll-off, тёплый свет не выжигается). `setPixelRatio(min(dpr,2))` — лимит для ретины iPad. Мебель кастует+принимает тень (placement.applyShadowFlags, пропуская прозрачное/шейдерное/свет/линии). Unlit-шейдеры (ТВ/аквалайзер/вода/окно) самосветящиеся — тон-маппинг их не трогает (пишут gl_FragColor напрямую).
- **ПРАВИЛО (свет = данные):** у каждого предмета в items.json есть поле `surface` (matte по умолчанию | emissive | reflective | transparent | refractive | absorbing | combined) — оптическая природа. Новый СВЕТЯЩИЙСЯ предмет (лава-лампа, неон, фигурка на стену) = запись в items.json с `surface:"emissive"` + `makeApplianceLight` в его builder-функции; риг и гашение по току уже готовы. См. PROGRESS.md «Идеи на потом» (играть со светом).
- **Ремонт (старт игры):** комната начинается голым бетоном (grid.js/walls.js), мебель в панели заблокирована (ui.setLocked). Записи placement:"reno" в items.json (паркет, обои) применяются кликом по ячейке (game.js → applyReno → applyParquet/applyWallpaper), после обоих мебель разблокируется. Подсказки ведут игрока по шагам.
- **Электричество:** розетка — настенный предмет с sockets:2 и fixedWallHeight:0.3 (вешается только у пола); приборы — поля cordLength (ТВ 2, магнитофон 2.5, торшер 3 клетки). power.js: автоподключение к ближайшей розетке со свободным гнездом в радиусе шнура, шнур рисуется линией по полу. Пересчёт — по onLayoutChange из placement.js. Будущее: удлинители и докупка розеток = новые записи в данных.
- **Бонусы за сочетания:** data/combos.json (декларативно, как items.json). Типы условий: placed, connected, near, onRug (combos.js). Имена — locales combos.<id>. Активные бонусы — списком под шкалой уюта. Уют = предметы + ремонт + бонусы + квесты (агрегация в game.js → refreshComfort).
- **Квесты (v0.2):** data/quests.json + движок src/quests.js (типы: placed, connected, near, nearWindow, nearWall, onRug c minCount, seatsNear). Активны первые 2 невыполненных по порядку из JSON; выполнение навсегда (не отменяется). Награды: comfort и/или items (предметы добавляются в панель через ui.changeCount). Журнал — #ui-quests слева под шкалой уюта (счётчик N/10 + активные). Тост о выполнении показывается через setTimeout(60), чтобы не был перетёрт хинтами установки.
- **GitHub:** репозиторий https://github.com/Olegk77777/novoselie (пока публичный, перед релизом закроем). Деплой: обычный `git push` в main → GitHub Pages обновляет игру на https://olegk77777.github.io/novoselie/ за 1–2 минуты. Файл .nojekyll в корне обязателен (отключает лишнюю обработку Pages).

## Как работаем
- **Маленькие шаги.** Один шаг = одна работающая фича. После каждого шага игра должна ОТКРЫВАТЬСЯ и работать. Сломалось — чиним до того, как идти дальше.
- **Git с первого коммита.** Инициализируй репозиторий, коммить после каждого рабочего шага с понятным сообщением на русском. Объясни Олегу один раз, что такое коммит и зачем.
- **Пушить сразу в main (постоянное разрешение).** Олег тестирует игру и на ПК, и на iPad через GitHub Pages — поэтому после каждого рабочего шага надо не только закоммитить, но и сразу выложить правки в `main` на GitHub (`git push`). Без пуша изменения до планшета не доезжают. Это durable-разрешение: push в этот репозиторий делать без отдельного спроса. **Нюанс worktree:** сессии Claude Code часто идут в отдельной ветке-worktree (`claude/...`), а Pages собирается ТОЛЬКО из `main` — значит пушить надо именно в main, а не в свою ветку. Когда HEAD на claude-ветке и она не разошлась с main: `git push origin HEAD:main`, затем подтянуть локальный main перемоткой (`git -C <главный-worktree> merge --ff-only <claude-ветка>`). Если main ушёл вперёд и fast-forward невозможен — не форсить, сказать Олегу. Перед пушем кода не забывать поднять версию кэша `?v=N` (см. раздел «Кэш / версионирование»).
- **Не усложнять.** Никаких лора, инвентарей, крафта, сохранений в облако и прочего, пока не закрыта текущая версия из CONCEPT.md. Если идея классная, но не в текущей версии — записывай в PROGRESS.md в раздел «Идеи на потом».
- **Перед концом сессии** — обнови PROGRESS.md: что сделали, что следующее, на чём остановились.
